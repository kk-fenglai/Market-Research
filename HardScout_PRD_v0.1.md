# HardScout — 产品需求文档（PRD）

> **一句话定位**：面向个人开发者与中小硬件公司的、**立项阶段**的 AI 竞品情报 agent——输入"我想做一个 XX 硬件"，输出竞品全景、差异点、使用场景地图与市场缺口，帮你在写第一行代码 / 画第一块板之前判断"该不该做、怎么差异化"。

## 0. 阅读说明（给未来的自己）

这份 PRD 有一个**贯穿全文的纪律**，决定产品的生死：

**本产品的核心是"立项决策级的竞品产品情报"。** 查料比价 / 选型 / 买货 / 附加服务这条价值链是**挂在决策报告后面的"最后一公里"**，用来放大报告价值、并承接变现——而**绝不是**一个让人来"搜零件、比价"的独立工具入口。

- **Last-mile（活）**：先帮你判断"这产品该不该做、差异化在哪" → 顺手给候选 BOM、报价、购买链接。值的不是比价数据本身（commodity），是它已在你的决策上下文里组装好。
- **Destination（死）**：做一个独立的查料/比价入口，正面拼 Octopart / 立创 / Flux.ai 免费且数据更全的事——必死。

**自检红线**：如果某次迭代发现自己在做一个"先搜零件"的入口，而不是"先出决策、再附器件"，停下来重读这一节。器件信息只能跟在决策后面，不能站在决策前面。

## 1. 背景与机会

### 1.1 问题

硬件 maker 和中小硬件公司在**立项阶段**面临一个高焦虑、低工具支持的决策：

- 我想做的这个硬件产品，市面上**谁已经在做**？
- 他们之间**怎么差异化**（规格、定价、形态、目标人群）？
- 各家打的是**什么使用场景**？
- **市场缺口**在哪？我进去的差异化切口是什么？
- 这事到底**该不该做**？

这个问题今天的解法是：手动 Google + 逛电商 + 翻 Reddit/论坛 + 拼 Excel，耗时数天到数周，且结论质量取决于个人调研能力。

### 1.2 现有工具为什么不解决它

| 工具类别 | 代表 | 它解决的问题 | 为什么不解决本问题 |
|---|---|---|---|
| 元器件搜索/选型 | Octopart、Findchips、立创、Flux.ai、EasyEDA | 查料、比价、选型、买货 | 只到"零件"层，不回答"产品/市场"层 |
| 专业拆解情报 | TechInsights、S&P/IHS、Caresoft | 竞品 BOM 级成本对标 | 年费数万刀，中小玩家碰不起；且是拆解，非立项决策 |
| 通用市场调研 | Statista、SimilarWeb 等 | 行业大盘数据 | 不垂直、不到产品颗粒度、不给差异化建议 |

**结论**：在"中小玩家可负担"× "产品/市场决策颗粒度"这个象限里，市场是空的。这就是 HardScout 的切口。

### 1.3 为什么是现在（Why now）

- LLM + 联网检索让"把一份立项竞品调研从几周压缩到几小时"在技术上成立。
- AI 选型/PCB 工具（Flux.ai 等）正在教育市场接受"AI 辅助硬件开发"，但它们集中在下游设计，上游决策仍是空白。

### 1.4 创始人不公平优势

- **领域知识**：在商米（POS / 零售硬件）做 EU 市场，懂硬件产品、认证、渠道、竞品格局——99% 的独立开发者没有。
- **可复用资产**：已有市场调研 agent 的 PRD 与代码基础。HardScout 本质是**把通用市场调研 agent 垂直化到硬件领域**，架构可大量复用，比从零做的人快一截。
- **技术栈成熟**：Next.js / Vercel / Supabase / Python·FastAPI / Claude·DeepSeek 已是熟练工具链。
- **跨境供应链优势**：中国背景 + 身处欧洲 + 硬件行业经验——能给欧洲 maker 提供"靠谱中国供应商 / 打样渠道"这种他们极难自己搞定、又愿意付钱的附加服务。这是后文"附加服务"变现的根基（见 §3.4、§10.1）。

> ⚠️ **合规红线（必须先处理）**：本产品与现任雇主（商米）所在行业高度相关。在投入前需自查：是否触及竞业条款、是否使用了任何雇主非公开信息、是否存在利益冲突。**本产品的所有数据必须来自公开来源，零雇主内部信息。** 建议把目标品类**避开商米直接竞品（POS/支付终端）**，选一个中性垂直起步（见 §4.3）。

## 2. 目标用户

### 2.1 核心用户画像（ICP）

不是"硬件大厂采购"（封闭、难触达、已被巨头工具服务），而是：

**Persona A — 独立硬件开发者 / Maker（验证期）**
- 有一个硬件产品想法（IoT 设备、消费电子小品类、开源硬件衍生品等）
- 想在投入前判断市场可行性与差异化空间
- 预算极敏感，付费意愿低，但量大、易触达（Reddit / HN / Hackaday / 嘉立创社区）
- 价值锚点：省下数天调研时间 + 避免做一个"已经有十家在做"的东西

**Persona B — 中小硬件公司的产品/创始人（立项期）— 主要付费方**
- 准备立项一个新硬件 SKU 或进入新品类
- 需要一份能拿去给团队/投资人看的、有结构的竞品与市场分析
- 有预算、付费意愿明确，但数量少、触达难度高于 A
- 价值锚点：决策质量 + 可交付的专业报告 + 持续盯竞品动态

> **战略取舍**：用 **A 做流量与口碑**，用 **B 变现**。免费/低价层服务 A，付费层服务 B。

### 2.2 明确不是目标用户

- 硬件大厂的专业情报/采购团队（已被 TechInsights 等服务）
- 只想"选型/买货"的人（用免费工具即可）
- 纯软件创业者

## 3. 价值主张与定位

### 3.1 定位陈述

> 对于**准备立项的硬件 maker 和中小公司**，HardScout 是一个**立项阶段的 AI 竞品情报 agent**，它能在几小时内给出竞品全景、差异化分析和市场缺口判断——不像 TechInsights 那样昂贵且偏拆解，也不像选型工具那样只停留在零件层。

### 3.2 核心价值（按重要性排序）

1. **决策**：该不该做、怎么差异化（painkiller，付费理由、产品的核心）
2. **效率**：数天调研 → 数小时
3. **可交付**：结构化、可分享、可拿去对内对外汇报的报告
4. **闭环**：从"想法 → 决策 → 候选 BOM → 报价 → 购买 → 供应链协助"一条龙——价值在于**整合与上下文**，不是单点比价（见 §3.4）

### 3.3 反定位（我们故意不做什么）

- 不做 PCB 设计 / EDA
- **不做独立的查料/比价入口**（destination），只做挂在决策报告后的器件延伸（last-mile，见 §0、§3.4）
- 不自建购物车 / 支付 / 当分销商——"买货"只是带 affiliate 的跳转
- 不承诺 BOM 级拆解精度（见 §9.3 诚实标注原则）
- 不做大厂级深度情报

### 3.4 完整价值链与四项能力定位

四项能力都**跟在决策报告后面**，按"AI 价值"和"变现潜力"分层对待，投入差别很大：

| 能力 | 定位 | 怎么做 | 投入 | 备注 |
|---|---|---|---|---|
| **选型** | 最值钱（AI 原生） | 从产品概念 → 候选 MCU/屏/传感器 + 取舍。这是 Octopart 做不到、LLM 擅长的跳跃 | 做深 | 报告的自然延伸 |
| **附加服务** | 最适合变现 | 供应商清单 / 打样对接 / BOM 成本估算——踩在跨境供应链优势上（§1.4） | 人工先行 | 第二条护城河 |
| **查料比价** | 薄便利层 | 直接调 Octopart/立创 API 显示报价库存 | 极低 | 不做入口，只在 BOM 行内显示 |
| **买货** | 薄便利层 | 带 affiliate 的分销商跳转链接 | 极低 | 顺带返佣收入，永不自建交易 |

## 4. 产品范围（Scope）

> 单人团队，范围纪律 = 生死线。MVP 只做"一条能跑通的价值主线"。

### 4.1 MVP 范围（Phase 1，In Scope）

- 用户用自然语言描述一个硬件产品想法
- Agent 产出一份**立项竞品情报报告**，含：
  - 竞品清单（5–15 个）+ 基本信息
  - 差异化对比矩阵（规格 / 价格 / 形态 / 目标人群）
  - 使用场景地图
  - 市场缺口与差异化切口建议
  - 数据来源与置信度标注
- 报告可在线查看 + 导出（PDF / Markdown）

### 4.2 明确 Out of Scope（MVP 不做）

- 独立的查料/比价入口（永不做，见 §0）
- 自建购物车 / 支付 / 交易（永不做）
- 持续监控 / 告警（Phase 2，用于解决复购）
- 团队协作 / 多人
- BOM 级成本估算（作为附加服务人工先行，不做自动化精算）
- 移动端 App

### 4.3 起步垂直品类（必须收窄）

不要做"通用硬件"。选 **1 个**有以下特征的垂直起步：
- 你有领域感、但**与商米无竞业冲突**（避开 POS/支付终端）
- 公开信息充足（电商、规格站、社区讨论多）
- 候选：消费级 IoT 设备 / 智能家居外设 / 开源硬件衍生品 / 特定传感器模组类产品

> 起步垂直的最终选择应由"那个愿意付钱的人"在哪个品类来决定（见 §15 开放问题）。

### 4.4 四项能力的分期落地（防 scope 膨胀）

四项**不能都进 MVP**，否则单人会淹死。按下表分期：

| 能力 | 进入阶段 | 形态 |
|---|---|---|
| 竞品决策报告（核心） | **MVP / Phase 1** | 产品主线，必须先跑通 |
| 选型（候选元件 + 取舍） | **Phase 1.5**（报告稳定后紧接着） | 报告的自然延伸，AI 生成 |
| 附加服务（供应链/打样/成本） | **Phase 0 即可人工试水**，Phase 2 产品化 | 先人工交付验证付费意愿 |
| 查料比价 | **Phase 2** | API 薄层，只在 BOM 行内显示 |
| 买货（跳转） | **Phase 2** | affiliate 链接 |

## 5. 核心功能与用户故事

### 5.1 用户故事

- **US-1**：作为一个 maker，我输入"我想做一个带 E-ink 屏的桌面天气站"，系统给我列出已有竞品，这样我能判断这个想法是否已经红海。
- **US-2**：作为一个中小公司产品负责人，我想要一张竞品差异化矩阵，这样我能向团队说明我们的切入点。
- **US-3**：作为用户，我想知道每条结论的来源和可信度，这样我能判断该信多少。
- **US-4**：作为用户，我想把报告导出成 PDF，这样我能拿去开会/给投资人看。
- **US-5（Phase 2）**：作为用户，我想订阅某个品类的竞品动态，每周收到新品/降价/新认证的变化，这样我能持续盯市场。

### 5.2 功能列表（MoSCoW）

| 优先级 | 功能 |
|---|---|
| Must（MVP） | 需求输入 → 竞品发现 → 差异化矩阵 → 场景地图 → 缺口建议 → 来源/置信度标注 → 报告查看/导出 |
| Should（Phase 1.5） | 选型：从想法生成候选元件 + 取舍（报告延伸）；附加服务的人工版（供应链/打样对接） |
| Could（Phase 2） | 查料比价（Octopart/立创 API，BOM 行内）、买货跳转（affiliate）、附加服务产品化、竞品持续监控+告警 |
| Won't（近期/永不） | 独立查料入口、自建购物车/支付、团队协作、移动 App |

## 6. 产品流程（Agent 工作流）

```
用户输入产品想法（自然语言）
      │
      ▼
[1] 需求解析 Agent
      - 抽取：品类、关键功能、目标人群、价位段、形态
      - 必要时反问澄清（最多 1–2 个问题）
      │
      ▼
[2] 竞品发现 Agent
      - 联网检索：电商 / 厂商站 / 规格站 / 社区
      - 产出候选竞品清单，去重、相关性过滤
      │
      ▼
[3] 信息抽取 Agent（对每个竞品并行）
      - 规格、价格、形态、宣传卖点、目标场景
      - 标注每条信息来源 URL + 置信度
      │
      ▼
[4] 综合分析 Agent
      - 差异化矩阵
      - 使用场景地图
      - 市场缺口 / 差异化切口建议
      │
      ▼
[5] 报告生成
      - 结构化报告 + 来源附录 + 置信度
      - 在线查看 / 导出
```

> 这套"解析→发现→抽取→综合→成稿"的多 agent 流水线，与你现有市场调研 agent 高度同构——**Phase 1 应直接 fork 复用，而非重写。**

## 7. 前端 UI 与界面设计

### 7.1 信息架构（整个产品只有三屏）

单人产品，IA 越简单越好。MVP 只有三个核心界面：**输入屏 → 生成屏 → 报告屏**。其余（仪表盘、多报告管理、协作）一律 Phase 2 之后再说。

```
[首屏·输入] ──提交想法──> [生成屏·等待态] ──完成──> [报告屏·核心交付]
```

### 7.2 首屏 · 输入（单一焦点）

- 中央一个大输入框，让人用自然语言描述硬件想法。
- 下方放 3–4 个示例 prompt 卡片降低启动门槛（"带 E-ink 屏的桌面天气站""宠物自动喂食器"…）。
- **不做**品类/价位等表单字段——那是反 AI 的。让 agent 自己解析，缺信息再反问 1–2 句（对应 §6 需求解析 Agent）。
- 这屏唯一的任务：让人三秒内敢开始。

### 7.3 生成屏 · 等待态（隐藏的信任关键）

报告需几十秒到几分钟，不能给个空转圈。要把 §6 的 agent 工作流**可视化成进度**：

```
解析需求 ✓  →  发现竞品 ✓  →  抽取信息（7/11）…  →  综合分析
```

- 每完成一步**流式吐出**已找到的竞品名，把"等待"变成"期待"。
- 作用不只是体验，更是**建立信任**：让人看见它在多源检索，而非凭空编造。

### 7.4 报告屏 · 核心（产品主菜，差异化所在）

自上而下布局：

1. **顶部**：查询的想法 + 生成时间 + 导出/重跑按钮。
2. **结论卡片（置顶，accent 描边）**：先给"该不该做 / 成熟度 / 切口"判断，不让人翻完数据自己悟。
3. **关键指标卡**：竞品数、价格区间、主场景。
4. **竞品差异化**：卡片列表，每个竞品挂**来源链接 + 置信度徽章**，直接竞品高亮。
5. **市场缺口卡（accent 底）**：明确给出你的差异化切口。
6. **底部**：置信度图例。

报告可在线查看 + 导出 PDF/Markdown（对应 §4.1）。界面示意见配套 mockup「HardScout 报告页」。

### 7.5 三条不可妥协的设计原则

1. **来源与置信度是一等公民，不是脚注**——这是区别于"AI 一本正经胡说"的护城河，视觉上必须重到无法忽略。
2. **结论永远前置**——用户要的是判断不是数据堆；每个区块先给结论再给依据。
3. **"推断"必须视觉隔离**——用独立颜色（紫）+ 图标标记 AI 估计项，让人一眼分清"查实" vs "推断"（对应 §9.3 诚实标注原则）。

### 7.6 设计语言与技术实现

- **栈**：Next.js + Tailwind + shadcn/ui，落地即用现有前端模板（对应 §8.2 技术栈）。
- **配色克制**：一个中性灰底 + 一个 accent 蓝；技术数据用等宽字体点缀。整体走 Linear / Vercel 那种技术工具感，不要消费级花哨。
- **信息密度**：仅报告屏高密度，其余界面极简。
- **置信度配色约定**：高 = 绿、中 = 琥珀、推断 = 紫。

### 7.7 MVP UI 范围纪律

MVP **不做**仪表盘、不做多报告管理、不做协作。UI 只需把"**输入 → 看着它干活 → 拿到可信报告 → 导出**"这条主线做到顺滑、可信，就足够。

## 8. 技术架构

### 8.1 复用决策（关键）

**直接复用现有市场调研 agent 代码库作为 Phase 1 骨架。** HardScout = 市场调研 agent + 硬件垂直的 prompt/schema/数据源适配。不重写编排层、检索层、报告层。

### 8.2 技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 前端 | Next.js + Vercel + Tailwind + shadcn/ui | 复用现有模板 |
| 后端 | Python + FastAPI | agent 编排 |
| 数据库 | Supabase | 用户、报告、缓存 |
| LLM | Claude（综合分析/成稿）+ DeepSeek（抽取/降本） | 双模型分工：贵的活给 Claude，量大的抽取给 DeepSeek |
| 检索 | Web search API（Perplexity / 自建检索） | 竞品发现与信息抽取 |
| 器件数据（P2） | Octopart/Nexar API、立创 API | 仅 Phase 2 收尾功能 |

### 8.3 模型分工原则

- **抽取类、高频、低创造性** → DeepSeek（控成本）
- **综合分析、报告成稿、需要判断力** → Claude
- 每份报告的 token 成本需在 §10 定价里算清（见单位经济）

## 9. 数据来源与策略（本产品的真正难点）

### 9.1 数据来源分层

| 数据 | 来源 | 可得性 | 准确度 |
|---|---|---|---|
| 竞品存在性、规格、定价 | 电商、厂商官网、规格站 | 高 | 中–高 |
| 使用场景、卖点 | 厂商文案、评测、社区 | 高 | 中 |
| 目标人群、口碑 | Reddit/论坛/评论 | 中 | 中 |
| 竞品内部用什么料（BOM） | 拆解 | **低** | 仅能推断 |

### 9.2 缓存与复用

- 同品类竞品数据缓存于 Supabase，跨用户复用，降低重复检索成本。
- 这同时是 Phase 2"持续监控"的数据底座。

### 9.3 诚实标注原则（产品信任的命根子）

- 每条关键结论附**来源链接 + 置信度（高/中/低/推断）**。
- "竞品用了什么料"一律标注为**推断**，绝不伪装成事实——这是与 TechInsights 的诚实分界，也是避免砸口碑的底线。
- UI 层面用独立颜色隔离"推断"项（对应 §7.5 原则 3）。

## 10. 商业模式与定价

### 10.1 复购问题（必须正面解决）

立项竞品分析是"一次性"行为，对订阅制是硬伤。两条对策：

1. **把一次性变成持续**：Phase 2 上线"竞品持续监控"——订阅某品类后，每周推送竞品新品/降价/新认证/新评测，把"立项工具"升级为"持续盯市场的雷达"。这是订阅制成立的关键。
2. **一次性也收费**：单份深度报告按次付费，服务"只立项一次"的用户。

### 10.2 定价结构（初版假设，待验证）

| 档位 | 对象 | 形态 | 价格（假设） |
|---|---|---|---|
| Free | Persona A 引流 | 限次/浅层报告 | ¥0 |
| Pay-per-report | 偶发需求 | 单份深度报告 | 按次（如 $19–49/份） |
| Pro 订阅 | Persona B | 多份报告 + 持续监控 | 月订阅（如 $29–99/月） |

> ⚠️ 上线前必须算**单份报告的 LLM+检索成本**，确保毛利为正。这是单人产品能否活下去的核心数字。

### 10.3 单位经济（待填）

- 单份报告成本 = 检索成本 + DeepSeek 抽取 + Claude 成稿 ≈ **[待测]**
- 定价需 ≥ 3× 边际成本

### 10.4 收入来源（多条腿，按可靠性排序）

1. **报告订阅 / 按次**（主收入）：核心决策报告，见 §10.2。
2. **附加服务**（高毛利、踩在跨境供应链优势上）：供应商清单、打样对接、BOM 成本估算——人工/半人工交付，可定高价，是第二条护城河。
3. **持续监控订阅**（Phase 2，解决复购）：竞品动态周报/告警。
4. **affiliate 返佣**（薄、被动）：买货跳转到 Octopart/立创/Mouser 的分销返佣。**注意**：返佣需要流量规模才有意义，早期可忽略，绝不为它牺牲产品定位。

> 关键判断：查料比价/买货**不单独收费**（免费工具遍地，付费意愿会蒸发），它们的作用是抬高报告价值；真正能溢价的是"决策报告 + 附加服务"。

## 14. 里程碑 / Roadmap

> **核心原则：先卖后建。** 不要先写代码。

### Phase 0 — 手工验证（0 代码，1–2 周）
- 对那个"愿意付钱的人"最关心的品类，**纯人工**扒数据 + 用 Claude 整理，交付 1 份高质量报告。
- 顺便试探：他是否愿意为**附加服务**（找供应商 / 打样对接）额外付费——这是最高毛利的一条腿，越早验证越好。
- 收钱、交付，观察其**是否愿意第二次/第二个月再付**。
- **门槛**：≥1 人连续付费 ≥2 次，才进入 Phase 1。否则止损。

### Phase 1 — MVP（fork 市场调研 agent）
- 复用现有 agent，做通"输入→竞品报告→导出"主线。
- 三屏 UI（输入 / 生成 / 报告）按 §7 落地。
- 限定 1 个垂直品类。
- 目标：把 Phase 0 的手工流程自动化，服务前 10–20 个用户。

### Phase 1.5 — 选型延伸 + 附加服务半自动
- 报告后接"候选元件 + 取舍"（AI 生成，§3.4）。
- 把 Phase 0 跑通的供应链/打样附加服务半产品化（仍可人工兜底）。

### Phase 2 — 器件薄层 + 订阅化
- 接 Octopart/立创 API，BOM 行内显示报价库存 + affiliate 买货跳转（薄层，§3.4）。
- 上线竞品持续监控 + 告警（解决复购）。
- 扩展第 2 个垂直品类。

### Phase 3 — 规模化
- 多品类、报告模板库、缓存数据资产化。

## 15. 开放问题（决策前必须回答）

1. **那个愿意付钱的人，想买的是哪一半？** 是"帮我搞清楚该不该进/怎么差异化"（painkiller，对），还是"帮我选料找货"（免费工具即可，付费会蒸发）？他具体怎么说的？

4. **单份报告成本是多少？** 决定定价与生死。
5. **这件事是否值得从 ExamPass 分走精力？** ExamPass 已建好、只差分销；HardScout 是新赌注。机会成本要算清。
